# FluidNotch (NotchScript) 项目开发白皮书

## 1. 项目愿景与定位

FluidNotch（NotchScript）旨在打造一款由 JavaScript/TypeScript 驱动的 macOS 灵动岛效率工具。项目以流体弹簧动效、苹果风格视觉体验和脚本化扩展能力为核心，面向希望快速构建个性化桌面微件与通知交互的开发者。

### 核心特色

1. **高完成度视觉与动效**
   - 追求接近 iOS 原生体验的非线性弹簧动画。
   - 强调流体形变、尺寸变化、圆角过渡和毛玻璃质感之间的同步。

2. **JavaScript 扩展能力**
   - 提供轻量级 JS SDK。
   - 支持通过 JavaScript 脚本和 JSON 配置动态注册 Widgets。
   - 允许第三方扩展定义图标、短文本、展开内容和交互行为。

3. **轻量开发链路**
   - 基于前端与 Node.js 技术体系构建。
   - 尽量减少对 Xcode 和 Swift 编译链路的依赖。
   - 优先支持快速试验、快速迭代和脚本化扩展。

## 2. 架构设计与技术路线

### 2.1 前端渲染层

#### 技术栈

- Electron
- HTML5
- CSS3
- Canvas 或 SVG

#### 窗口特性

灵动岛窗口需要具备以下基础特征：

- 无边框：`frame: false`
- 透明背景：`transparent: true`
- 永远置顶：`alwaysOnTop: true`
- 根据场景设置合适的置顶层级，例如 `screen-saver`

为避免常驻窗口干扰菜单栏和桌面操作，窗口在非交互状态下应支持鼠标穿透：

```js
islandWindow.setIgnoreMouseEvents(true);
```

当灵动岛进入展开或可交互状态时，再动态关闭鼠标穿透。

### 2.2 动画与视觉规范

#### 动画引擎候选

- Anime.js
- Framer Motion

#### 弹簧参数基线

以下参数可作为早期调试基线，后续需结合实际渲染效果校准：

| 参数 | 建议值 |
| --- | --- |
| Stiffness | `300` |
| Damping | `28` |
| Mass | `0.8` |

#### 视觉细节

- 展开和收起时，宽度、高度与 `border-radius` 应保持同步变化。
- 背景可采用毛玻璃与高饱和度叠加效果：

```css
backdrop-filter: blur(25px) saturate(180%);
```

- 边缘可使用轻量半透明描边，提升层次与精致感：

```css
border: 0.5px solid rgba(255, 255, 255, 0.15);
```

### 2.3 核心逻辑与桥接层

#### 技术栈

- Node.js 运行时
- Electron IPC

#### 进程通信

主进程与渲染进程之间通过 Electron IPC 传递状态、通知事件和 Widget 数据。

#### 扩展机制

项目应提供独立的扩展加载与执行机制：

- 通过沙盒环境运行外部 `.js` 脚本。
- 可评估使用 Node.js `vm` 模块实现基础隔离。
- 通过 `manifest.json` 描述扩展元数据和展示能力。

扩展清单可覆盖以下信息：

- 图标
- 短文本
- 展开态内容
- 基于 JSON 结构描述的卡片数据
- 后续可扩展的交互入口与权限声明

## 3. 核心场景：微信消息提醒

由于 macOS 微信缺少稳定的官方开放接口，早期实现可优先验证以下两条路径。

### 3.1 方案 A：系统日志流监听

通过 Node.js `child_process` 调用系统日志流命令，监听通知中心相关事件：

```sh
log stream --predicate 'subsystem == "com.apple.notificationcenterui"'
```

在日志流中筛选 `com.tencent.xinWeChat` 相关标识，尝试提取通知中的发件人和消息内容，并将事件推送到 FluidNotch。

#### 适用价值

- 非侵入式。
- 有机会快速验证“收到消息后自动展开灵动岛”的 MVP。

#### 风险与待验证项

- 系统日志字段和可见性可能随 macOS 版本变化。
- 通知内容能否稳定提取，需要在目标系统版本上验证。
- 需要评估隐私、权限和分发场景下的可用性。

### 3.2 方案 B：本地 WebHook 联动

借助具备 WebHook 能力的微信扩展，将消息推送到 FluidNotch 本地 HTTP 服务。

可由 FluidNotch 提供轻量接收端，例如基于 Express 的本地路由，用于接收消息事件并触发 UI 更新。

#### 适用价值

- 响应链路更直接。
- 适合验证低延迟通知交互。

#### 风险与待验证项

- 依赖第三方扩展能力和兼容性。
- 需要评估注入式扩展带来的安全、维护和分发风险。

## 4. 开发路线图

### Phase 1：基础骨架

- 搭建 Electron 项目骨架。
- 创建常驻刘海条窗口。
- 实现基础展开、收起和弹簧动画。
- 建立初版视觉参数基线。

### Phase 2：通知 MVP

- 验证系统日志流监听方案。
- 跑通微信消息触发链路。
- 实现“消息到达后灵动岛自动展开”的最小可行版本。

### Phase 3：扩展 API

- 抽象 Widget 扩展 API。
- 实现 `manifest.json` 读取与校验逻辑。
- 支持加载第三方 JavaScript 扩展脚本。
- 明确扩展权限、沙盒边界和数据协议。

## 5. 开发备忘

开始编码时，可优先根据本文档生成以下基础文件：

- Electron 主进程入口，例如 `main.js`
- 渲染层页面，例如 `index.html`
- 初版样式与动画模块
- Widget 扩展清单示例

项目早期应优先验证两件事：

1. macOS 刘海区域附近的窗口呈现与交互体验是否自然。
2. 通知触发链路是否稳定、可维护且符合后续分发要求。
